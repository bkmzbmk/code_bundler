# Code Bundler

Инструмент для сборки исходников C++/Python проекта в **один текстовый блок**
с метками начала/конца каждого файла — для удобной вставки в чат с ИИ.

Главная фишка — **анализатор зависимостей**: вы отмечаете нужные файлы, а
программа сама подтягивает связанные с ними (по `#include "..."` и
`import` / `from ... import`) до заданной глубины.

---

## Зачем это нужно

1. **Не таскать 100+ файлов вручную.** Отметил нужное — собрал — скопировал.
2. **Точный отбор под контекст ИИ.** Вместо «всего проекта» — только нужные
   файлы и их зависимости, чтобы не тратить лимит токенов впустую.

---

## Возможности

- 🌳 **Дерево файлов** проекта с чекбоксами (символы ☑/☐).
- 📁 **Выбор корневой папки** проекта.
- 🔗 **Анализатор зависимостей** (C++ и Python) с ограничением глубины.
- 🔢 **Подсчёт токенов** через `tiktoken` (+ грубая оценка, если не установлен).
- ⚠️ **Настраиваемый лимит токенов** с предупреждением о превышении.
- 🚫 **Игнор-паттерны** (`build/`, `.git/`, `__pycache__/` и т.п.) +
  поддержка корневого **`.gitignore`**.
- ✂️ **Опционально**: удаление комментариев и/или пустых строк.
- 📋 **Предпросмотр** результата и **копирование** в буфер обмена.

---

## Формат вывода

Каждый файл оборачивается фиксированными метками:

```
// ===== FILE START: относительный/путь/файла =====
<содержимое файла>
// ===== FILE END: относительный/путь/файла =====
```

Пути в метках — **относительные** от корня проекта, с разделителем `/`.

## Требования

- **Python 3.10+** (используется синтаксис `str | None`, `dict[...]`).
- **Tkinter** — обычно идёт в комплекте с Python.
  - Linux (Debian/Ubuntu): `sudo apt install python3-tk`
- **tiktoken** — *опционально*, для точного подсчёта токенов.
  Без него используется грубая оценка (≈4 символа на токен).

Установка `tiktoken` (по желанию):

```bash
pip install tiktoken
```

## Установка и запуск

```bash
git clone <repo-url>
cd code_bundler
python main.py
```

> На Windows, если `python` не найден, попробуйте `py main.py`.

## Как пользоваться

1. **«Выбрать папку»** — укажите корень проекта. Дерево заполнится файлами
   кода (папки из игнор-списка и `.gitignore` отсеиваются автоматически).
2. **Отметьте файлы** кликом по строке (☐ → ☑).
   Клик по папке отмечает всё её содержимое каскадно.
   Клик по значку `+`/`-` — раскрывает/сворачивает папку.
3. Задайте **«Глубину зависимостей»**:
   - `0` — только выбранные файлы;
   - `1` — + их прямые зависимости;
   - `2`, `3`, … — глубже.
4. Нажмите **«Подтянуть зависимости»** — программа доотметит связанные файлы.
5. При необходимости включите **«Удалять комментарии»** /
   **«Удалять пустые строки»** и задайте **«Лимит токенов»**.
6. Нажмите **«Собрать»** — результат появится в предпросмотре, снизу —
   счётчик токенов (краснеет при превышении лимита).
7. **«Копировать»** — bundle уходит в буфер обмена. Вставляйте в чат с ИИ.

## Как работает анализ зависимостей

Пути к заголовкам/модулям **разрешаются парсингом дерева проекта**
(без чтения include-путей из CMake — так решено в проекте).

**C++** (`#include "..."`, системные `<...>` игнорируются):
1. относительно папки текущего файла;
2. относительно корня проекта;
3. поиск по имени файла (basename) во всём дереве.

**Python** (`import` / `from ... import`, разбор через `ast`):
- относительные импорты (`from .`, `from ..`) — от папки файла;
- абсолютные — сопоставление `a.b.c` с `a/b/c.py` или пакетом
  `a/b/c/__init__.py` внутри проекта;
- внешние библиотеки (не найденные в дереве) — игнорируются.

Обход графа — **BFS** с ограничением по глубине.

## Поддерживаемые расширения

- **C/C++**: `.cpp`, `.cc`, `.cxx`, `.c`, `.h`, `.hpp`, `.hxx`, `.hh`
- **Python**: `.py`, `.pyw`

Изменить набор можно в `config.py` (`CPP_EXTENSIONS`, `PYTHON_EXTENSIONS`).

---

## Игнорирование файлов

- **Встроенные паттерны** — `DEFAULT_IGNORE` в `config.py`
  (`build`, `.git`, `__pycache__`, `venv`, `node_modules`, …).
- **`.gitignore`** — если он есть в корне проекта, его правила применяются
  автоматически (поддержаны `!`, `/`, `dir/`, glob `*`/`?`).
  Вложенные `.gitignore` в подпапках **не** читаются — только корневой.

---

## Конфигурация (`config.py`)

| Константа                  | Назначение                                   |
|----------------------------|----------------------------------------------|
| `FILE_START_TEMPLATE` / `FILE_END_TEMPLATE` | Шаблоны меток файлов        |
| `CPP_EXTENSIONS`, `PYTHON_EXTENSIONS`       | Расширения по языкам        |
| `CODE_EXTENSIONS`          | Итоговый набор расширений кода                |
| `DEFAULT_IGNORE`           | Игнорируемые папки/файлы по умолчанию         |
| `DEFAULT_TOKEN_LIMIT`      | Лимит токенов по умолчанию (120 000)         |
| `DEFAULT_TIKTOKEN_MODEL`   | Кодировка tiktoken (`cl100k_base`)           |
| `FALLBACK_CHARS_PER_TOKEN` | Символов на токен при оценке без tiktoken     |
| `FILE_ENCODINGS`           | Кодировки, перебираемые при чтении файлов     |

## Архитектура

Приложение построено по паттерну **MVP** (Model–View–Presenter):

- **Model** не зависит от GUI, доступен через фасад `AppModel`.
- **View** общается с **Presenter** только через интерфейс `IView` (ABC).
- **Presenter** не знает про Tkinter — связывает View и Model.
- Анализаторы языков — наследники `IDependencyAnalyzer` (легко расширяется).

### Структура проекта

```
code_bundler/
├── main.py                        # точка входа: собирает MVP и запускает
│
├── config.py                      # константы: метки, лимиты, расширения
│
├── model/
│   ├── file_node.py               # FileNode — узел дерева файлов
│   ├── project_scanner.py         # ProjectScanner — сканирование папки
│   ├── ignore_rules.py            # IgnoreRules — игнор-паттерны
│   ├── gitignore_rules.py         # GitignoreRules — разбор .gitignore
│   ├── file_reader.py             # read_text() — чтение с перебором кодировок
│   ├── bundler.py                 # Bundler — сборка текста с метками
│   ├── token_counter.py           # TokenCounter — tiktoken + fallback
│   ├── profile_storage.py         # ProfileStorage — заглушка (профили)
│   ├── dependency_graph.py        # DependencyGraph — граф + BFS
│   ├── app_model.py               # AppModel — фасад над всем Model
│   └── analyzers/
│       ├── base_analyzer.py       # IDependencyAnalyzer (интерфейс)
│       ├── cpp_analyzer.py        # CppAnalyzer (#include)
│       ├── python_analyzer.py     # PythonAnalyzer (import)
│       └── analyzer_registry.py   # выбор анализатора по расширению
│
├── presenter/
│   └── main_presenter.py          # MainPresenter (связка View<->Model)
│
├── view/
│   ├── iview.py                   # IView — интерфейс View (ABC)
│   └── main_view.py               # MainView (Tkinter)
│
└── test/
    ├── test_gitignore.py          # тест правил .gitignore
    ├── test_stage1.py             # ручной тест ядра (сборка)
    └── test_stage2.py             # тест анализатора зависимостей
```

## Тесты (ручные)

Запускать из корня проекта.

**Правила `.gitignore`** (без файловой системы):
```bash
python test/test_gitignore.py
```

**Сборка ядра** (Этап 1):
```bash
python test/test_stage1.py <путь_к_проекту>
```

**Анализатор зависимостей** (Этап 2):
```bash
python test/test_stage2.py <путь_к_проекту> <rel/path/to/file> [depth]
# пример:
python test/test_stage2.py ./myproj src/main.cpp 2
```

## Известные ограничения

- Удаление комментариев — простые regex-эвристики (не полноценный парсер);
  комментарии внутри строковых литералов не учитываются. Для отправки ИИ
  это приемлемо.
- `.gitignore`: `**` трактуется как обычный `*`; `*` матчит и `/`;
  читается только корневой `.gitignore`.
- C++ include-пути разрешаются эвристически (без CMake include dirs).
  При нескольких файлах с одинаковым именем выбирается наиболее подходящий
  по хвосту пути.
